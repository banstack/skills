Coffee Shop Ratings

A lightweight site where visitors browse a fixed list of coffee shops and leave anonymous star ratings and comments — no account, no server, no database.

Problem Statement

Coffee shop visitors have no lightweight way to see what nearby regulars think of a shop, or to leave their own quick impression. Existing review platforms require an account, bury ratings under unrelated content, and are heavier than the decision actually warrants — "is this place good?" shouldn't take a sign-up.

Solution

A single-page site listing a fixed set of local coffee shops, each showing an average rating at a glance. Opening a shop shows its individual ratings and comments, and lets any visitor submit a 1–5 star rating with an optional comment and display name — no login required. All data is stored in the browser via a mock backend, since this build exists to exercise the design process, not to ship a production service.

User Stories

  1. As a visitor, I want to see a list of nearby coffee shops, so that I can decide where to go.
  2. As a visitor, I want to see each shop's average rating at a glance, so that I can quickly compare options.
  3. As a visitor, I want to see how many ratings a shop has received, so that I can judge how reliable the average is.
  4. As a visitor, I want to open a shop's detail page, so that I can read individual ratings and comments.
  5. As a visitor, I want to submit a star rating from 1 to 5, so that I can share my experience.
  6. As a visitor, I want to optionally add a comment with my rating, so that I can explain what I liked or didn't.
  7. As a visitor, I want to optionally add a display name, so that my feedback feels personal without requiring an account.
  8. As a visitor, I want to submit a rating without logging in, so that leaving feedback is frictionless.
  9. As a visitor, I want to see my rating appear immediately after submitting, so that I know it was received.
  10. As a visitor, I want ratings to persist across page reloads in my browser, so that my feedback isn't lost.
  11. As a visitor, I want ratings ordered newest first, so that I see recent feedback first.
  12. As a visitor, I want the star input to be a simple click or tap, so that rating takes seconds.
  13. As a visitor, I want to see a shop's neighborhood, so that I know roughly where it is.
  14. As a visitor, I want the site to work with no real backend, so that it runs as a self-contained demo.
  15. As an internal reviewer, I want the ratings data layer isolated behind one interface, so that tests and any future backend swap don't touch UI code.

System Design Decisions

Modules

seedData

Static fixture of ~8–10 coffee shops (id, name, neighborhood, description). Not user-editable.

ratingsStore

The one seam. Wraps localStorage; owns all reads and writes of ratings.

CoffeeShopList

Renders seeded shops with each one's computed average rating.

CoffeeShopDetail

Shows one shop's ratings/comments and hosts the submission form.

RatingForm

Star picker plus optional name/comment inputs; calls ratingsStore.addRating.

Data Flow

seedData → CoffeeShopList static shop list, rendered with computed averages
ratingsStore.getRatingsForShop() → CoffeeShopList / Detail per-shop ratings, used to compute averages and render the list
RatingForm submit → ratingsStore.addRating() validates stars 1–5, writes to localStorage
addRating() return value → CoffeeShopDetail re-render component renders from the returned list — never re-reads storage itself

Interfaces

  • getShops(): Shop[] Returns the fixed seeded shop list — id, name, neighborhood, description.
  • getRatingsForShop(shopId: string): Rating[] Returns all ratings persisted for a shop, newest first.
  • addRating(shopId: string, input: { stars: 1|2|3|4|5, comment?: string, displayName?: string }): Rating[] Validates stars is 1–5, generates an id and timestamp, persists to localStorage, returns the shop's updated Rating[].

Decisions

localStorage instead of a real server

The goal of this build is a self-contained demo the pipeline can exercise end to end. A real server adds process and deployment concerns that don't serve that goal.

One seam (ratingsStore) instead of separate read/write seams

Reads and writes both touch the same persistence concern. A single module keeps the tested interface small without fragmenting into mocks that mirror the implementation rather than the behavior.

Compute averages on read, don't store a running average

Ratings volume is small and local. Recomputing from raw ratings avoids any risk of a stored average drifting from the underlying data.

Testing Decisions

ratingsStore

Tests exercise ratingsStore's public functions only — getShops, getRatingsForShop, addRating — asserting on return values, never reaching into localStorage directly or asserting on component internals. This means storage could later be swapped for a real API without invalidating a single test.

Prior art

None — this is a fresh test build with no existing codebase. The ratingsStore tests are the first written, following one-seam vertical slices per the tdd skill.

Out of Scope

  • User accounts or login
  • Adding new coffee shops — the shop list is fixed and seeded
  • Editing or deleting existing ratings
  • A real backend server or database
  • Photo uploads for shops or reviews
